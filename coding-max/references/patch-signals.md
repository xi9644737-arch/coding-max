# 终审信号

这些是调查入口，不是单独定罪标准。必须结合契约、数据流和回归测试判断。

## 补丁信号

| 信号 | 需要证明 |
|---|---|
| 新增 `if`/默认值 | 校验是否位于拥有该契约的边界，而非掩盖上游坏数据 |
| 新增 `try/catch` | 异常是否被处理/传播并保留上下文，是否改变调用方契约 |
| 改常量 | 值是否来自规范、配置或测试边界 |
| 改参数默认值 | 调用方遗漏是否才是真正根因 |
| diff 很大 | 是否混入无关重构，能否缩到影响面 |

输入校验、错误翻译和兼容默认值可能正是根因修复；不要因语法形状误报。

## 认知红旗

- “应该就是”：先验证假设。
- “先修后补测试”：先取得 RED。
- “手动看过”：保留可重复命令与结果。
- “再试一次”：同视角失败后换数据流/调用链/配置/时序。
- “全量红所以目标没修好”：先区分当前回归与既有基线。

## Premortem

- 破坏：返回格式、副作用、异常类型、锁序或持久化状态会否变化？
- 回滚：能否单独 revert？是否存在迁移或中间状态？
- 边界：至少三个，含一个非 happy path。
- 模式：同模块和相邻模块是否有同根因？
- 依赖：调用方和外部消费者是否兼容？

## 架构信号

| 信号 | 本次策略 |
|---|---|
| 改动跨 3+ 不相关模块 | 标 `arch-coupling`，先做安全止血 |
| 循环依赖 | 标 `circular-import`，独立规划依赖反转 |
| 逻辑重复 5+ 处 | 扫描全部，按授权决定统一修复或记录债务 |
| 业务逻辑在 UI/API 层 | 标 `wrong-layer`，不要在 Bug 修复中擅自重分层 |
| 单文件职责混杂 | 标 `god-module`，不要以行数单独定罪 |

## 反事实

临时 revert 核心修复时，回归测试应重新失败；恢复修复后应通过。若结果不变，测试或根因至少有一个无效。新增测试通常增加代码量，因此“净删除大于净新增”不能作为根因修复的必要条件。
