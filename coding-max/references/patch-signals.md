# 补丁信号 + 认知红旗 + 反事实 + 架构异味

## 补丁信号 (代码层 — 步骤7自检用)

修复中出现以下任一模式→是创可贴，非根因修复:

| # | 信号 | 为什么是创可贴 | 正确做法 |
|---|------|---------------|---------|
| 1 | 只加了 if-else | 没处理根因，在外面包了层壳 | 追踪异常值的来源，从源头消除 |
| 2 | 只包了 try-catch | 错误被藏了不是被解了 | 确定异常条件，消除异常触发路径 |
| 3 | 只改了数字/常量 | 不知道为什么是这个值 | 解释为什么这个值，写入注释或配置 |
| 4 | 只加了默认参数 | 没追踪调用方为什么不传 | 跟踪所有调用方，修复遗漏传参 |
| 5 | 修复解释不了根因 | 靠运气编程 | 说不清为什么能修好=没修好 |

## 认知红旗 (思维层 — 铁律1触发条件)

修复过程中出现以下念头→**立刻停止**，回到5 Whys换视角:

| 红旗 | 现实 | 正确做法 |
|------|------|---------|
| "先快速修一下，后面再深入看" | 后面永远不会来 | 现在深入，或标记临时方案+注明过期时间 |
| "我试试改 X 看看效果" | 你在调试不是修复 | 先理解为什么X应该是某值，再改 |
| "跳过测试，手动验证一下就行" | 手动验证不可重复不可审计 | 写自动化测试 |
| "应该就是 X 的问题" | "应该"≠确认 | 先验证假设再下手 |
| "不太确定但这样应该能行" | 不确定=没挖到根因 | 回到诊断阶段换视角 |
| "再试一次"(已2+次) | 碰运气 | 硬停，展示所有尝试，请用户介入 |

## 反事实测试 (步骤7自检 — 三层自检之反事实)

```
1. 脑内 revert 这个 diff
2. 原来的 bug 会重现吗？

会 → 根因修复 ✅
不会 → 可能是创可贴 → 检查:原来的代码跟这个bug无关?改动多余?
```

## 架构异味 (arch-信号 — 触发Standard模式)

这些不是创可贴，但意味着根因不在代码层面:

| 信号 | 说明 | 策略 |
|------|------|------|
| 修复需改3+个不相关模块 | 改动扩散=架构耦合 | 最小止血+标记`arch-coupling` |
| 两个模块互相导入 | 循环依赖 | 延迟导入让bug不触发+独立重构任务 |
| 同一逻辑5+处重复 | 缺少抽象 | 只修触发位置+建议抽取 |
| 业务逻辑在UI/API层 | 层级混乱 | 最小修复+标记`wrong-layer` |
| 单文件400+行含不相关函数 | 上帝模块 | 不在此bug中拆分+标记`god-module` |
| 修改A总牵连B | 紧耦合 | 标记`tight-coupling`+建议解耦迭代 |

## 根因修复特征 (对照检查)

- 能从根因出发解释"为什么之前会出bug"
- 同类模式在项目中全部被发现和处理
- 有回归测试，且测试在未修复版本上确实失败
- 修复后代码比修复前更简单(净删除>净新增)
- (架构根因)最小安全修复不改变架构结构

## Premortem 检查清单 (步骤4，修复方案写好后、写代码前)

### 破坏 — 会搞坏什么?
- [ ] 如果修复引入新bug，最可能的症状?
- [ ] 现有调用方中哪些会因行为变化出错?(grep调用点)
- [ ] 是否有隐式依赖(返回值格式/副作用/异常类型)被此修改改变?

### 回滚 — 炸了怎么退?
- [ ] 修复可否单commit revert而无额外清理?
- [ ] 是否涉及DB迁移/配置文件变更?(不可逆→单独标注)
- [ ] revert后是否有中间状态需手动处理?

### 边界 — 极限情况
- [ ] 空输入/零值/负数/超长字符串/Unicode/None/并发
- [ ] ≥3个边界用例，≥1个非happy-path

### 模式 — 同类还有哪?
- [ ] `grep`搜索同样反模式(裸except/`.get(`无默认值/全局可变变量)
- [ ] 同模块+相邻模块已检查
- [ ] 是否本次一并修?→冲击波评估(步骤5)

### 依赖 — 谁在调用?
- [ ] `grep -r "函数名"`列出所有调用方
- [ ] 每个调用方是否兼容新行为?
- [ ] 是否有外部API消费者无法同步更新?
