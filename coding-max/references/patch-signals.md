# 5个补丁信号 + 6个认知红旗 + 反事实测试

## 补丁信号（代码层）

| # | 信号 | 为什么是创可贴 | 正确做法 |
|---|------|---------------|---------|
| 1 | 只加了 if-else | 没处理根因，在外面包了层壳 | 找到为什么会有异常值，从源头修复 |
| 2 | 只包了 try-catch | 错误被藏了不是被解了 | 确定什么情况会抛异常，消除异常条件 |
| 3 | 只改了数字/常量 | 不知道为什么是这个值 | 解释为什么是这个值，写进注释或配置 |
| 4 | 只加了默认参数 | 没追踪调用方为什么不传 | 跟踪所有调用方，修复遗漏的传参 |
| 5 | 修复解释不了根因 | 靠运气编程 | 说不清为什么能修好 = 没修好 |

## 认知红旗（思维层）

修复过程中出现以下任一念头 → 大概率在自欺欺人：

| # | 红旗念头 | 现实 | 正确做法 |
|---|---------|------|---------|
| 1 | "先快速修一下，后面再深入看" | 后面永远不会来 | 现在就深入，或者标记为临时workaround（注明过期时间） |
| 2 | "我试试改 X 看看效果" | 你在调试不是在修复 | 先理解为什么 X 应该是某个值，再改 |
| 3 | "跳过测试，我手动验证一下就行" | 手动验证不可重复、不可审计 | 写自动化测试，不然下次回归没人知道 |
| 4 | "应该就是 X 的问题，直接修 X" | "应该" ≠ 确认 | 先验证假设，再写修复 |
| 5 | "不太确定但这样应该能行" | 不确定 = 没挖到根因 | 回到步骤2，换视角重新分析 |
| 6 | "再试一次"（已2+次） | 你在碰运气，不是修bug | 停止，展示所有尝试，让用户介入 |

## 反事实测试

```
1. 脑内 revert 这个 diff
2. 原来的 bug 会重现吗？

会 → 根因修复 ✅
不会 → 可能是创可贴，需要进一步检查
      原来代码没问题 → 这次改动是多余的
      或者原来代码也有问题但和这个bug无关
```

## 架构异味信号（架构层）

这些不是创可贴，但意味着 bug 的根因不在代码层面：

| 信号 | 说明 | 策略 |
|------|------|------|
| 修复需改 3+ 个不相关的模块 | 改动扩散 = 架构耦合 | 最小安全修复 + 标记 arch-coupling |
| 两个模块互相导入 | 循环依赖 | 延迟导入让bug不触发 + 独立重构任务 |
| 同一逻辑在 5+ 处重复 | 缺少抽象 | 只修当前触发位置 + 建议抽取 |
| 业务逻辑出现在 UI/API 层 | 层级混乱 | 最小修复 + 标记 wrong-layer |
| 单文件超过 400 行且包含不相关函数 | 上帝模块 | 不在此bug中拆分 + 标记 god-module |
| 修改 A 总是牵连 B | 紧耦合 | 标记 tight-coupling + 建议解耦迭代 |

## 根因修复的特征

- 能从根因出发解释"为什么之前会出bug"
- 同类模式在项目中全部被发现和处理
- 有回归测试，且测试在未修复版本上确实会失败
- 修复后代码比修复前更简单（净删除 > 净新增）
- （架构根因时）最小安全修复不改变架构结构，重构 = 独立任务

## Premortem 检查清单（步骤3.1，修补前）

写下修复方案后、写代码前，逐条回答：

### 破坏 — 这个修法会搞坏什么？
- [ ] 如果修复引入了新 bug，最可能的症状是什么？
- [ ] 现有调用方中哪些会因行为变化而出错？（看 git grep 调用点）
- [ ] 是否有隐式依赖（返回值格式/副作用/异常类型）被此修改改变？

### 回滚 — 炸了怎么退？
- [ ] 修复是否可以单 commit revert 而无需额外清理？
- [ ] 是否涉及数据库迁移/配置文件变更？（不可逆变更需单独标注）
- [ ] 如果 revert，是否有中间状态需要手动处理？

### 边界 — 极限情况想过没？
- [ ] 空输入 / 零值 / 负数 / 超长字符串 / Unicode / None / 并发
- [ ] 至少列出 3 个边界用例，其中至少 1 个是非 happy-path

### 模式 — 同类问题还有哪？
- [ ] `git grep` 搜索同样的反模式模式（如 `except:` / `.get(` 无默认值 / 全局可变变量）
- [ ] 同模块 + 相邻模块均已检查
- [ ] 是否需要在本次一并修复？→ 冲击波评估（步骤3.2）

### 依赖 — 谁在调用这个？
- [ ] `grep -r "函数名"` 列出所有调用方
- [ ] 每个调用方是否仍兼容新行为？
- [ ] 是否有外部 API 消费者无法同步更新？
