# 安全解耦

只在耦合已有行为/维护风险证据时读取。目标是缩小变化半径并恢复所有权，不是追求某种分层教条。

## 1. 保护现有行为

- 为用户可见行为建立 characterization test；它锁定当前事实，不代表当前设计正确。
- 为将被抽出的协议、状态变化和错误语义建立 contract test。
- 记录当前可重复命令、输入、预期、实际和已知基线失败。
- 无可信测试入口且风险不低时，先调用 pipeline；不得用“编译通过”代替行为保护。

## 2. 定义最小目标

只回答本次迁移需要的问题：

- 业务规则、状态和错误语义分别由谁拥有？
- 哪个稳定接口隔离易变实现？
- 允许的依赖方向是什么？
- 第一个可迁移 seam 在哪里？
- 哪些行为、数据格式和副作用必须保持？
- 每一步怎样回滚？

不要因预期复用创建框架。抽象必须消除已证明的重复知识、逆向依赖或状态所有权冲突。

## 3. 选择结构手术

- **入口重复规则**：抽到共同 use case；入口只做解析、授权、调用和展示。
- **核心依赖具体 adapter**：由核心定义 port，外层实现 adapter，并在 composition root 注入。
- **重复协议/枚举**：建立一个权威契约，迁移消费者后删除副本。
- **共享可变状态**：指定单一所有者，让写入、失效、锁和生命周期从该边界发出。
- **时序耦合**：把隐式顺序变为显式状态机、事务或资源生命周期；验证非法顺序。
- **god module**：沿已有变化原因和契约 seam 分离；不按行数机械拆文件。
- **循环依赖**：先找双向知识，移动契约或反转一侧依赖；禁止用延迟 import 隐藏结构问题。

## 4. 增量迁移

1. 建新 seam，但不复制业务规则。
2. 迁移一个强相关调用方。
3. 运行 characterization、contract 和该调用方测试。
4. 检查错误、状态、副作用和依赖方向。
5. 继续下一调用方；每步保持可回滚。
6. 最后切断旧入口，删除双写/双读、wrapper、fallback 和废弃配置。

兼容层只有在外部消费者无法同步迁移时保留，并必须写所有者、退出条件、验证方式和最迟删除触发器。

## 5. 验证与反事实

- 原用户行为保持或按授权契约变化；目标 Bug 仍由 coding-max 验证。
- 新边界可独立测试，旧方向被测试或静态检查阻断。
- 临时撤销关键迁移时，边界测试应重新失败；否则测试或根因无效。
- 检查至少三个边界以及并发/锁/缓存/持久化中与本次相关的一项。
- 搜索旧符号、重复常量、死分支和临时诊断；检查无残留调用方。

若迁移连续三次失败、每一步暴露不同共享状态，或需要同时重写多个不相关子系统，停止并返回 coding-max 记录恢复点/架构阻塞，不继续叠加适配层。

## 交回

向 coding-max 提供：结构根因、迁移范围、行为保护、目标方向证据、删除内容、验证命令和剩余风险。不得关闭原 Bug 或替代最终 Review。
