# 耦合审计

只在 Explore、Guard 或需要证明结构性根因时读取。目标是找出“为什么一个局部变化会迫使无关部分一起变化”，不是绘制漂亮架构图。

## 证据分类

| 类型 | 可验证信号 | 常见误报 |
|---|---|---|
| 结构依赖 | 循环 import、下层反向依赖入口层、调用方读取内部字段 | 同层内聚模块间的正常调用 |
| 契约耦合 | 协议、枚举、错误翻译或业务规则复制多份 | 纯展示文案或语言边界适配 |
| 共享状态 | 多模块无所有者地读写全局/缓存/配置，写入后失效路径不明 | 明确封装且只读的不可变配置 |
| 时序耦合 | 初始化、调用、锁或清理顺序改变即失败 | 明确事务或协议要求的顺序 |
| 变更耦合 | 一个规则反复迫使不相关文件同步修改，同类缺陷跨入口复发 | 一次性迁移或机械版本升级 |
| 测试耦合 | 模块只能通过整套系统验证，替换一个 adapter 需启动全部依赖 | 本来就必须端到端验证的系统边界 |

## 审计步骤

1. 锁定一条用户行为链：入口、use case、规则/状态所有者、adapter、副作用和返回契约。
2. 读取真实调用方、import、配置、状态读写、错误格式和相关测试；排除 vendor、生成物与缓存。
3. 最多提出三个可证伪假设，例如“UI 复制规则导致 change coupling”，不要把工具指标直接写成结论。
4. 用工作示例对照坏路径；必要时查看版本历史中哪些文件经常因同一规则共同变化，但 co-change 只作线索。
5. 检查至少三个边界：正常路径、一个错误/空值边界、一个状态/顺序/并发边界。
6. 扫描相邻同模式，区分局部缺陷、重复结构和一次性迁移债务。

## 判定门槛

至少满足一项才进入 Untangle：

- 可复现行为错误能追到结构边界；
- 同根因缺陷已在两个以上入口/模块出现；
- 局部规则变化必须修改无关模块且不存在权威契约；
- 状态或调用顺序没有唯一所有者并能造成错误；
- 依赖方向阻止目标模块独立验证、替换或回滚；
- 临时兼容路径已形成双写、双读或永久分叉风险。

只存在命名不美、文件较长、import 较多或抽象层较少时，不得判为结构缺陷。

## 输出

记录范围、实际依赖方向、证据、被破坏边界、影响、目标方向、最小 seam、行为保护缺口和剩余未知。Explore 到此停止；确认缺陷后交 coding-max 建档，或在活动 Bug 中继续 Untangle。
