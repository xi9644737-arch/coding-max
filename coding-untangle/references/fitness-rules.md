# 架构疫苗

只在目标边界已由实际缺陷或迁移证明后读取。先定义最小可执行疫苗，再由现有测试/lint 实现；需要搭建或接入 CI 时交给 pipeline。

## 选择规则

从已确认根因反推一条最低成本规则：

| 根因 | 优先疫苗 |
|---|---|
| 逆向依赖/循环 | import/依赖方向测试 |
| 多入口复制规则 | contract test 证明入口调用共同 use case |
| 重复协议/枚举 | 单一权威定义与重复扫描 |
| 共享可变状态 | 所有者 API、写入/失效/锁行为测试 |
| 时序耦合 | 状态转换和非法顺序测试 |
| 永久兼容层 | 旧符号/路径禁止测试和删除条件 |

优先复用项目已有测试、lint、类型或依赖检查工具。不要为一条规则引入重型框架，也不要以复杂度阈值代替真实边界。

## Pipeline 交接

交接内容必须包含：

- 已确认的根因和报告路径；
- 禁止/允许的依赖或行为；
- 扫描范围与排除目录；
- 旧代码下应失败的 RED 示例；
- 修复后应通过的命令；
- 本地/CI 预期和已知基线失败。

Pipeline 只把规则实现为测试、lint、pre-commit 或 CI gate，不得自行决定架构。远程未实际成功时只能写待验证。

## 验收

疫苗必须在旧结构或定向变异下失败、在目标结构下通过；错误信息应指出被破坏边界和修复方向。它不得扫描 vendor/生成物，不得依赖特定 Agent、模型或 IDE，也不得把警告冒充阻断。
