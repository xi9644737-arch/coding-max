# 安全退役与发布清场

## 1. 基线和候选

记录任务根、目标版本、dirty baseline、允许范围及排除目录。读取已有 `PROJECT_PROFILE.md` 作为候选和验证路由，不替代当前源码证据。候选包括旧入口、兼容协议、feature flag、迁移 wrapper、废弃配置、重复资产、过时测试、debug/trace 和临时发布物。生成物、缓存和用户数据先确认所有者，不默认删除。

分类：

- `safe-delete`：已有替代物或确定不再需要，消费者证据完整。
- `deprecate`：仍有公开或外部消费者，需要窗口、告警或迁移说明。
- `retain`：当前行为、构建或恢复仍需要。
- `blocked`：无法证明、权限不足或验证链不可信。

## 2. 死亡证明

“搜索零引用”只是假设。逐类检查：

- 直接 import、调用、继承、注册表和测试；
- 动态加载、反射、字符串路由、插件发现、环境变量与配置键；
- CLI/GUI/API、定时任务、hook、脚本和操作手册；
- 构建清单、打包资源、安装器、发布产物和运行时入口；
- 公开 API、序列化字段、数据库/持久化格式、迁移与旧客户端；
- 文档、示例和病历是否会诱导未来重新实现旧路径。

对替代物做反事实：临时恢复旧入口或移除替代物时，合同测试/门禁应暴露差异。结果不变时继续调查，不能伪造证明。

## 3. 兼容窗口

内部且可证明无消费者的路径可直接退役。公开 API、持久化数据、跨版本协议或第三方入口必须满足其一：

1. 已完成声明过的 deprecation 窗口；
2. 有迁移工具、版本检测和回滚路径；
3. 用户明确授权破坏性变化并记录影响。

不把永久兼容当安全。每个 shim/flag 必须有 owner、退出条件和目标版本；到期未满足则写 `blocked`。

## 4. 增量退役

按一个强相关集合执行：保护替代行为 → 删除实现 → 删除调用/注册 → 删除只服务旧路径的测试、配置和文档 → 运行定向验证。测试若同时保护新合同则保留或改写，不与实现一起删除。

不得移入 `.attic`、graveyard、backup、legacy-copy 等目录；Git 历史与墓碑已提供恢复证据。未知二进制、许可证、生成源和用户文件默认保留并报告。

发现 Bug 时停止该候选并交 coding-max；需要改变分层或状态所有者时交 coding-untangle；没有可执行门禁时先交 coding-pipeline。

## 5. 发布验证

按项目实际选择：目标测试、全量测试、lint/format/typecheck、build/package/install/start、CLI/GUI 主入口、迁移/回滚、发布清单和 secret/debug 扫描。未运行的远程 CI、安装平台或生产观察写 `unknown`，不能推断通过。

最后搜索旧符号、协议字段、配置键、文档名称和复制实现；命中必须分类为允许引用、墓碑证据或未清残留。退役改变入口、命令、结构或关键路径时，以已验证事实更新 `PROJECT_PROFILE.md`；未变则不写。只有替代路径可用、旧路径不可达、验证真实通过时写 `verified`。
